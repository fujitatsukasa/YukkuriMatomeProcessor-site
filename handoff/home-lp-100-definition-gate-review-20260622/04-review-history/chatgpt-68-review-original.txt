# 総評と現状スコア

## **68 / 100**

**前回 64点 → 今回 68点（+4点）**

安全修正の方向は正しいです。特に、未検証の「実操作」「完成映像」「60秒操作デモ」を削除し、動画を「制作フロー解説」と明示したこと、`VideoObject`を撤去したこと、FAQの表示内容とJSON-LDを一致させたこと、モバイルHeroに商品画像を入れたことは、明確な改善です。

ただし、**39,800円の商品を販売するLPとしては、まだ公開完成状態ではありません。**

最大の理由は、コピーではなく次の3点です。

1. **スクリーンショットが説明内容を証明していない**
2. **Free / Premium、PC台数、更新範囲など、購入判断に必要な条件が未確定**
3. **Home以外の料金・FAQ・共通コンポーネントに旧来の強い表現が残っている**

現状は、Homeだけを慎重に直したため、ページ内の誤認リスクは下がりました。しかし、サイト全体で見ると、料金ページなどから再び「制限解除」「YMM4へ直反映」「動画作成導線」といった表現に戻ります。**Homeの安全修正が購入導線全体に波及していません。**

また、同梱証跡では配布情報に以下の不一致があります。

* LP側：`0.0.18`、約296.3MB
* 配布証跡側：`0.0.35`、Setup.exe 301,620,172 bytes
* 最終公開ゲート：`Passed: false`
* Authenticode、署名済みパッケージ、署名前提のFresh VM確認：未通過

したがって、現状のHomeに多数ある**実行ファイル直リンクCTAは、いったん `/download/` の確認ページ経由に変えるべき**です。

### 公開判断

| 対象                   | 判定                     |
| -------------------- | ---------------------- |
| Homeの情報提供ページとしての公開   | 条件付き可                  |
| Homeからの実行ファイル直接配布    | P0修正前は非推奨              |
| Premium 39,800円の積極販売 | 購入条件確定前は不可             |
| 架空数値を入れず安全コピーだけ追加修正  | 可能だが、75点前後で頭打ち         |
| 90点以上                | 実アプリ証拠・契約条件・配布情報の確定が必要 |

なお、同梱ソースはフルリポジトリではなく、package scriptsが参照する一部スクリプトやlockfile等を含んでいません。そのため、提供された実行結果は確認しましたが、全コマンドの独立再実行まではしていません。

## UI・レスポンシブ・アクセシビリティ判定

| 確認項目                    | 判定        | 所見                                                            |
| ----------------------- | --------- | ------------------------------------------------------------- |
| 390×844の初回表示に商品画面が入るか   | **△**     | 技術的には入る。ただし画像内UIが小さく、証拠内容を判読できない                              |
| Heroリードが長すぎないか          | **×**     | 1文に入力・選定・話者・改行・素材・YMM4・最終作業まで詰め込みすぎ                           |
| 画像が切れていないか              | **○ / △** | `contain`で切れは解消。ただし縦長画面では大きな余白が生じる                            |
| sticky CTAが邪魔にならないか     | **△**     | 非表示時のDOM削除は良い。表示時は約1行分以上を覆うため、最終行やLightboxとの干渉を追加検査すべき        |
| 料金比較がモバイルで読めるか          | **×**     | 980px以下でヘッダー行を非表示にするため、各値がFreeかPremiumか視覚的に分からない              |
| FAQのキーボード利用             | **○**     | native `<details>/<summary>`は良い                               |
| Lightboxのキーボード利用        | **△**     | Escape、初期フォーカス、復帰は実装済み。ただし`img[tabindex="-1"]`までフォーカス候補に含めている |
| reduced-motionで情報が消えないか | **概ね○**   | transition/animationを無効化している。ただし実ブラウザ設定での表示検査を追加すべき           |
| 200%拡大に耐えるか             | **未確認**   | 現テストのroot 18pxは200%拡大の代替にならない                                 |
| 320 CSS px相当でreflowするか  | **未確認**   | 横スクロール検査はあるが、400%相当のreflow検査ではない                              |

WCAGでは、テキストを200%まで拡大しても情報・機能が失われないこと、320 CSS px相当で二方向スクロールを必要としないreflowが求められます。現在の18pxテストでは不足です。([W3C][1])

また、操作はキーボードから利用でき、意味のあるフォーカス順序を保つ必要があります。Lightboxは基礎実装が良い一方、フォーカス対象の抽出と背景の操作抑止をもう一段厳密にすべきです。([W3C][2])

## SEO・構造化データ判定

| 項目                            | 判定         | 改善案                                              |
| ----------------------------- | ---------- | ------------------------------------------------ |
| `SoftwareApplication`と可視コンテンツ | **一部一致**   | publisher、価格、OSは一致。descriptionとdownloadUrlは見直し必要 |
| `FAQPage`と表示FAQ               | **一致**     | 現在の10件との機械的一致は修正済み                               |
| `VideoObject`を出さない判断          | **妥当**     | 現動画が実操作・完成証拠ではないため、現段階では出さない                     |
| publisher                     | **適切**     | `OTM株式会社`でよい                                     |
| price / priceCurrency         | **適切**     | Free 0、Premium 39800、JPYは事実と一致                   |
| operatingSystem               | **適切**     | Windows 10 / 11                                  |
| softwareRequirements          | **不足**     | x64、WebView2、インターネット、Googleログイン等を正本確定後に追加        |
| downloadUrl                   | **要修正**    | 配布情報不一致が解消するまで直接EXEではなく安定した`/download/`を推奨       |
| title                         | **改善余地あり** | 「YMM4向け台本・素材準備ツール」より、入力と前準備の関係を具体化               |
| description                   | **過大**     | 「AI台本生成、YMM4反映まで」は現証拠より強い                        |
| OG画像                          | **弱い**     | 実画面の証拠が小さく、商品特性が一目で分からない                         |

GoogleのSoftwareApplication向けリッチリザルトでは、`name`、`offers.price`等に加え、リッチリザルト適格性のために`aggregateRating`または`review`が要求されています。評価を捏造してまで適格化してはいけません。現在は、正確な意味付けを優先すべきです。([Google for Developers][3])

`VideoObject`の削除は正しい判断です。Googleは、ユーザーが実際に動画を視聴できるページ上で、名称、サムネイル、公開日などの正確な情報を要求しています。([Google for Developers][4])

さらに、Googleは2026年5月7日以降、FAQリッチリザルトを表示しない方針に変更しています。したがって、現在のFAQPage整合は品質管理としては良いものの、検索結果の表示強化にはなりません。保守コストを下げるため、FAQPage JSON-LDを残す積極的理由がなければ撤去しても構いません。([Google for Developers][5])

1. Hero
2. 実画面で見る3工程

6. 必須環境と責任分担
