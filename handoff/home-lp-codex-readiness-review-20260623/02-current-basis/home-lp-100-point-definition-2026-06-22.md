# Home / 販売LP 100点定義 v2 2026-06-22

## 1. 目的

この文書は、ゆっくりまとめプロセッサーのHome、料金、購入、配布、FAQ、法務、サンプルを含む販売導線について、何を満たせば100点と呼ぶかを固定する採点基準です。

100点は、普遍的・永久的な「完璧」を意味しません。

次の組合せに対して、定義済みの完了条件をすべて満たした状態を意味します。

- サイトcommit
- アプリ公開バージョン
- 配布manifest
- 料金・利用条件
- 法務文書バージョン
- 証拠素材バージョン
- 検証実施日

これらのいずれかが変更された場合、影響する項目を再評価します。

## 2. 評価対象

評価対象は、見た目だけのLPではありません。

以下を含む「販売開始可能性を含むLP完成度」を評価します。

1. 商品とプランの事実
2. LP上のコピーと情報設計
3. 実画面・出力・動画等の証拠
4. 購入・権限付与・返金
5. 配布・署名・更新
6. 利用規約・プライバシー・特商法表示
7. UX・アクセシビリティ
8. SEO・schema・性能・技術品質
9. QA・変更管理

採点前に、D1で次を確定します。

- 対象ユーザー
- Homeの主要CV
- 副次CV
- LPが販売中ページか、公開前案内ページか

## 3. 100点の定義

100点とは、次がすべて成立した状態です。

1. 販売に使う商品事実が承認済みである
2. すべての販売主張に根拠がある
3. 表示内容と実アプリ挙動が一致している
4. Home、料金、購入、配布、FAQ、法務、アプリ内表示が一致している
5. 購入から権限付与まで実環境で完了する
6. Free版の配布から起動まで実環境で完了する
7. 重大な誤認、無権利素材、危険な導線がない
8. 定義済みのアクセシビリティ・性能・QA条件を満たす
9. 未確定事項が販売コピーやCTAに漏れていない
10. すべての判断に責任者、根拠、適用版が記録されている

## 4. 二層評価

評価は、次の二層で行います。

### 4.1 販売開始ゲート

- Go
- Conditional Go
- No-Go

### 4.2 品質スコア

- 0〜100点

ゲート判定は品質スコアより優先します。

高得点でも、販売開始ゲートがNo-Goなら販売を開始しません。

## 5. ハードゲート

### G0 事実・権利・法務ゲート

次のいずれかがあればNo-Goです。

- 重大な虚偽または事実矛盾
- 未確認値の確定表現
- 無権利のスクリーンショット、動画、素材
- 法務ページと販売コピーの重大な不一致
- 商品の範囲を超える自動化・成果保証表現
- ユーザーに重大な誤認を与えるCTA

G0が不合格の場合、点数は公開せず「採点保留」とします。

### G1 Premium購入ゲート

以下がすべて満たされた場合のみ `purchaseReady` をtrueにします。

- 価格、税込表示、通貨
- 買い切り・月額なし
- FreeとPremiumの権利差
- Premium上限・公正利用
- PC・アカウント・再認証
- 更新・保守範囲
- 支払方法
- 最終確認画面
- 権限付与方法と提供時期
- 返金条件
- 利用規約・特商法表示・プライバシーの整合
- 実決済またはテストモードによるE2E確認
- 法務・事業責任者の承認

`purchaseReady` は手入力せず、上記条件から算出します。

### G2 Free配布ゲート

以下がすべて満たされた場合のみ `downloadReady` をtrueにします。

- 公開バージョン
- 配布チャンネル
- ファイル名
- ファイルサイズ
- SHA-256
- 署名状態
- 発行者名
- SmartScreen実測結果
- 配布URL
- リリースノート
- クリーンWindows 10 / 11での起動確認
- 更新または再取得手順
- 公開責任者の承認

`downloadReady` は手入力せず、上記条件から算出します。

### G3 証拠ゲート

主要な販売主張ごとに証拠を用意します。

証拠は次のいずれでも構いません。

- 実データ入りアプリ画面
- 実際の出力ファイル
- 入力から出力までの操作記録
- 実YMM4画面
- 再現可能なテスト結果
- 利用許諾済みサンプル

実操作動画は必須ではありません。

架空UI、AI生成UI、実画面風モックは商品証拠として使用しません。

### G4 品質検証ゲート

以下を完了します。

- lint
- typecheck
- build
- コピー横断検査
- broken link検査
- meta / JSON-LD検査
- desktop / mobile視覚QA
- 200%文字拡大
- 320 CSS px相当のreflow
- キーボード操作
- フォーカス管理
- prefers-reduced-motion
- 性能予算
- 購入E2E
- 配布E2E
- 代表ユーザーによる主要タスク確認

## 6. 100点ルーブリック

| 区分 | 配点 | 評価対象 |
| --- | ---: | --- |
| 商品・プラン事実 | 15 | 対象ユーザー、入力、出力、非対応範囲、Free、Premium、動作環境、ライセンス、更新 |
| 販売導線・購入判断 | 15 | 主要CV、CTA分岐、比較、購入不安、購入・配布・権限付与E2E |
| 証拠・主張立証 | 15 | 主張証拠対応、実画面、実出力、サンプル、素材権利、鮮度 |
| コピー・情報設計 | 15 | 初回理解、短さ、具体性、重複排除、全ページ整合、期待値調整 |
| 法務・契約・データ・配布安全性 | 15 | 支払、提供、返金、利用規約、特商法、プライバシー、外部送信、署名 |
| UX・レスポンシブ・アクセシビリティ | 10 | reflow、200%、キーボード、意味構造、コントラスト、動画、操作性 |
| SEO・schema・性能・技術品質 | 10 | title、description、canonical、OG、schema、性能、build、リンク |
| QA・トレーサビリティ・運用保守 | 5 | 根拠、承認者、適用版、テスト、変更時の再評価 |
| 合計 | 100 |  |

## 7. 採点方法

各区分を原子的なチェック項目へ分割します。

各チェック項目は次のいずれかで採点します。

- 0%: 未提供、矛盾、根拠なし
- 50%: 安全側に処理済みだが、未承認または未検証
- 100%: 証拠付きで承認済み、公開表示と一致し、必要なE2E検証済み

自由裁量による中間点は付けません。

同じ不備を複数区分で重複減点しません。
ただし、ハードゲートの上限は別途適用します。

## 8. 点数上限

最終点は、加重点数と該当上限の小さい方とします。

| 状態 | 上限 |
| --- | ---: |
| G0不合格 | 採点保留 / No-Go |
| 商品・プラン・購入条件が未確定 | 79 |
| 配布条件が未確定でFree取得が主要CV | 79 |
| 条件は確定したが主要主張の証拠が不足 | 89 |
| 証拠は揃ったがE2E・アクセシビリティ・性能検証が未完了 | 94 |
| 全ゲート合格 | 100まで |

## 9. スコア帯

| 点数 | 状態 |
| --- | --- |
| 0〜59 | 根本修正が必要 |
| 60〜79 | 安全な下書き・公開前案内 |
| 80〜89 | 条件付き販売候補 |
| 90〜94 | リリース候補 |
| 95〜99 | 重大問題のない販売開始候補 |
| 100 | 定義済み条件をすべて満たした状態 |

ゲート判定がNo-Goの場合、点数帯にかかわらず販売開始しません。

| D2 | Free機能、上限、単位、周期、上限到達時挙動 |
| D3 | Premium権利、上限、公正利用、警告・停止条件 |
| D4 | Googleアカウント、PC台数、同時利用、端末変更、再認証 |
| D5 | 価格、決済、最終確認、権限付与、失敗時対応、返金、領収書 |
| D6 | 更新、保守、サポート、メジャー版、EOL |
| D7 | 対応入力、対応URL、取得範囲、非対応条件、権利確認 |
| D8 | YMM4対応版、出力形式、受渡手順、動作環境、互換性 |
| D9 | 外部AI/API、送信データ、保持・削除、分析・テレメトリー |
| D10 | 配布版、ファイル、SHA、署名、SmartScreen、リリースゲート |

各決定には次を記録します。

- status
- decision
- owner
- sourceRefs
- approvedBy
- approvedAt
- effectiveFrom
- validForAppVersion
- validForSiteCommit
- reviewTriggers

## 11. 正本の優先順位

商品事実の根拠は次の順で扱います。

1. 事業責任者・法務責任者による承認済み決定記録
2. 実アプリ、実配布物、決済、権限付与の一次証拠
3. 署名済み法務文書
4. リリースmanifestと互換性テスト結果
5. `product-facts.ts`
6. LP、FAQ、料金、ブログ等の表示

AGENTS.md、LP本文、FAQ本文だけを商品事実の一次証拠にしません。

`product-facts.ts` 以外に、公開条件、価格、版番号、SHA、上限等を重複ハードコードしません。

## 12. 担当範囲

### 事業責任者

- 対象ユーザー
- 主要CV
- Free/Premium条件
- ライセンス
- 更新・サポート
- 販売・配布開始承認

### 法務・データ責任者

- 利用規約
- 特商法表示
- 返金
- プライバシー
- 外部送信
- 素材権利

### アプリ・リリース責任者

- 実機能
- 対応URL
- YMM4互換性
- 配布版
- 署名
- クリーン環境確認

### Codex

- データ構造
- 表示実装
- ゲート計算
- 横断整合
- 自動テスト
- 視覚QA
- レビューパック作成

Codexは未確認の事業判断、法務判断、素材権利を確定しません。

## 13. 100点までの進行順

1. この定義を承認する
2. D1〜D12の責任者と根拠形式を決める
3. D1〜D11を証拠付きで確定する
4. `product-facts.ts` へ反映する
5. `purchaseReady`、`downloadReady`、`publishReady` を自動算出する
6. Home、Purchase、Download、FAQ、Samples、法務を横断更新する
7. 実画面・実出力・証拠を反映する
8. 購入・権限付与・返金・配布をE2E確認する
9. アクセシビリティ・性能・SEO・schemaを検証する
10. 最終レビューと公開責任者の承認を行う
11. サイトcommit、アプリ版、法務版を記録して採点する

## 14. 100点完了条件

次をすべて満たした場合のみ100点とします。

- G0〜G4がすべて合格
- すべての原子的チェック項目が100%
- D1〜D11が証拠付きでconfirmed
- D12の全ゲートがtrue
- 未確認事実が本文、meta、JSON-LD、alt、aria-label、title属性にない
- 全ページと実アプリの説明が一致
- 実購入、権限付与、返金、配布の検証記録がある
- 実素材の権利・出典・撮影元版が記録されている
- 法務・事業・リリース責任者が承認している
- 評価対象のcommit、アプリ版、法務版、検証日が記録されている

## 15. 実装前の必須質問

各回答には、決定者、根拠資料、適用アプリ版、適用サイトcommit、発効日を付けます。

1. Homeの主要CVは何ですか。Free版取得、Premium購入、問い合わせのどれを最優先にしますか。
2. 現在の公開版が実際に出力・準備できる成果物は何ですか。CSV、`.ymmp`、素材一覧、台本などを列挙してください。
3. Freeで利用できる機能と上限は何ですか。単位、リセット周期、上限到達時の挙動も必要です。
4. Premiumで追加される権利と残る制約は何ですか。公正利用、停止・警告条件を含みます。
5. GoogleログインはFreeでも必須ですか。オフライン利用、ログアウト時、アカウント削除時の挙動はどうなりますか。
6. ライセンスの端末条件は何ですか。登録台数、同時利用、PC変更、再認証、VM、チーム利用を決めてください。
7. 買い切りに含まれる更新・保守範囲は何ですか。メジャー版、セキュリティ修正、対応終了条件を含みます。
8. 対応するURL・入力形式と、非対応条件は何ですか。ログイン必須、非公開、ペイウォール等を含めてください。
9. 対応YMM4バージョンと受渡形式は何ですか。動作確認済み組合せと非互換時の扱いも必要です。
10. 必須動作環境は何ですか。x64、WebView2、通信、空き容量、管理者権限等を確認してください。
11. 購入フローの正式仕様は何ですか。決済方法、税表示、最終確認、領収書、権限付与時間、失敗時対応を含みます。
12. 返金条件の正確な文章は何ですか。7日以内であれば常に返金なのか、重大不具合時のみなのかを明確にしてください。
13. 外部AI/APIへ何を送信しますか。送信先、送信項目、保存期間、削除方法、利用者への表示を決めてください。
14. 公開配布の正本は何ですか。バージョン、ファイル名、サイズ、SHA、署名者、SmartScreen実測、公開ゲート結果が必要です。
15. 証拠として掲載を承認できる実素材はどれですか。撮影元版、サンプルデータの権利、マスキング、掲載期限を確認してください。
