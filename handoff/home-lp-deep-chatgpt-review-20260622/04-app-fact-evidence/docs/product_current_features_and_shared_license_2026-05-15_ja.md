# Yukkuri Matome Processor Ver.2 現行機能と共有ライセンス案

作成日: 2026-05-16
対象時点: 2026-05-15 時点のリポジトリ上の実装・設計資料・UI構成
目的: ローカルMCP対応AIなどに読み込ませて、買い切り価格、販売ページ、機能訴求、ライセンス設計を検討するための整理

## 0. ローカルMCP対応AIに読ませる時の前提

この資料では、Yukkuri Matome Processor Ver.2 の現行機能と、今後追加したい「1ライセンスを別PCまたは別編集者に貸し出せる仕組み」を分けて整理する。

重要な区分は次の通り。

- 「現行機能」は、リポジトリ上に実装・UI・設計資料・検証資料として存在するもの。
- 「共有ライセンス案」は、現在の課金・権限・ログイン基盤に追加する提案仕様であり、現時点で完成実装として売ってよい機能ではない。
- 販売文面では「完全自動で動画が完成する」とは言わない。正しくは「テンプレート運用、台本作成、素材整理、YMM4反映準備、外部AI連携を通じて制作時間を短縮するソフト」と表現する。
- YMM4連携は強力だが、実機互換性・公開配布前検証が残る領域があるため、「YMM4の編集準備・テンプレ反映を支援する」と表現するのが安全。
- 外部AI連携は、YMPが外部AIのAPIキーを預かって勝手に使う方式ではない。Claude Desktop、Claude Code、Codex、Cursor、ローカルMCP対応CLI などのホスト側に生成を任せ、YMPはプロンプト生成、検証、変換、プレビュー、承認付き反映を担当する。

## 1. 製品の一言説明

Yukkuri Matome Processor Ver.2 は、まとめ動画・ゆっくり動画・VOIRO系動画・解説動画の制作を支援する制作補助ソフト。

主な価値は、記事や掲示板素材の取得、AI台本作成、台本編集、テンプレート設計、YMM4向けプロジェクト準備、サムネ用AIプロンプト作成、外部AI連携、診断・サポートを一つの制作フローにまとめること。

特に売りになるのは、単体の台本生成ツールではなく、制作現場で時間を取られる「素材取得」「台本整形」「キャラ別セリフ化」「テロップ分割」「テンプレ反映」「YMM4連携」「外部AIへの手渡し」「サムネ制作準備」までをつなげている点。

## 2. 想定ユーザー

- ゆっくり実況・ゆっくり解説・まとめ動画を作る個人制作者
- 5ch、掲示板、ブログ、ニュースなどをもとに台本を作る編集者
- YMM4を中心に動画制作している人
- 動画制作を外注編集者と分担したいチャンネル運営者
- Claude Desktop、Claude Code、Codex、Cursor、ローカルMCP対応CLI などのローカルMCP対応AIを制作に使いたいが、プロンプト作り、JSON整形、YMM4投入、承認管理まで手でやるのが面倒な人
- サムネイルの構成、訴求、AI画像生成プロンプトまでまとめて作りたい人

## 4. ランチャー・アカウント・課金・アップデート

### 4.1 ランチャーの役割

ランチャーは、ユーザーが最初に触る入口。

主な役割は次の通り。

- アプリの起動
- ログイン状態の確認
- ライセンス状態の確認
- 決済画面への誘導
- 利用権限の同期
- アップデート確認
- メインアプリ起動
- エラーや診断情報の表示

ランチャー側には、Standard月額、Premium月額、Premium年額、Premium買い切りの選択肢がUI上に存在する。

### 4.2 現行のプラン概念

現行の課金・権限設計では、おおむね次の区分がある。

- Free
- Standard / Pro相当
- Premium
- Premium Lifetime / 買い切り

現行資料・実装上の利用枠は次の通り。

Free:

- 台本取得: 初回10回
- 台本生成: 初回3回
- デフォルト台本編集プリセット利用

Standard / Pro相当:

- Freeの内容を含む
- 台本取得: 月50回
- 台本生成: 月20回
- 出力文字数目安: 2000文字
- 台本編集プリセットのカスタマイズ

Premium:

- Standardの内容を含む
- 台本取得: 月200回
- 台本生成: 月60回
- 出力文字数目安: 5000文字
- AIモード・台本作成機能
- YouTube API検索・保存
- Agent Bridge系の高度な外部AI連携
- 優先サポート

Premium Lifetime / 買い切り:

- Premium相当の買い切り権限として扱う設計がある
- lifetimePlan が premium_lifetime の場合、Premium相当権限になる
- 台本取得・台本生成は無制限扱いの設計がある

注意点:

- 買い切りプランのUI・権限データ構造は存在する。
- ただし、Stripe Checkout の汎用購入タイプとして Premium買い切り・年額が完全に通っているかは実装確認が必要。
- シリアルコード機能は現状 standard_monthly と premium_monthly が中心で、premium_lifetime のシリアル発行は未完成扱いが安全。

### 4.3 権限で開放される主な機能

Premiumで強く訴求できる権限は次の通り。

- 台本取得・台本生成の上限拡張または無制限化
- AI台本作成
- Agent Bridge / MCP連携
- 台本プリセットのカスタマイズ
- サポート優先度

設計思想:

- Web UI内のプロジェクト/ボードを制作の中心データとして扱う。
- YMM4は最終的な出力・プレビュー先として扱う。
- 手でYMM4へ全部打ち込む工程を減らす。

### 9.3 MCPの安全設計

Agent Bridgeは、拒否が標準の安全設計。

MCPから到達できないもの:

- 課金の直接変更
- 管理者操作
- シリアル発行
- 生のAPIキー
- Firebaseトークン
- Stripeシークレット
- 任意ファイル読み取り
- 任意ファイル削除
- 任意コマンド実行
- 生SQLite/生datデータ
- デバッグ設定の勝手な変更
- 任意URLからの無許可画像取得
- Google認証情報

承認が必要なもの:

- CSV書き出し
- Project.json作成
- ボード反映
- YMM4同期
- 外部取得の一部
- ローカルに影響する操作

承認方式:

- ローカルユーザーの明示承認
- confirm_token による一回限りの承認
- 外部AIが勝手に破壊的操作をできないようにする

### 9.4 ローカルMCP対応AIとの関係

外部AI連携の標準ルートはローカルMCPです。

その場合の流れ:

- YMPがプロンプトパッケージを作る
- ローカルMCP対応AIがYMP toolを呼ぶ
- AIがJSONや台本を返す
- YMPが構造検証、整形、プレビュー、反映を行う

つまり、YMPは「ローカルMCP経由で制作データに変換する受け皿」として価値を持つ。

## 12. YMM4連携・テンプレート運用

YMPはYMM4向け制作補助が大きな強み。

### 12.1 YMM4連携の考え方

基本設計:

- YMPのWeb UI内プロジェクト/ボードを制作の中心にする
- YMM4は出力・プレビュー・最終編集先として扱う
- YMPからYMM4へ片方向に反映する
- YMM4側で手動編集した内容をYMPへ完全自動で逆同期する設計ではない

### 12.2 できること

- CSVからYMM4向けプロジェクト作成
- Project.jsonプレビュー
- Project.json作成
- `.ymmp`ファイルからフォーマット追加
- YMM4キャラクター集の保存
- YMM4キャラクター情報の読み取り
- 立ち絵パス一括変更
- テンプレート層の読み取り
- キャラ、字幕、画像、音声、動画、図形アイテムの扱い
- YMM4反映前のプレビュー
- 承認付きYMM4同期
- YMM4の準備状態確認
- YMM4が準備できていない場合は失敗させる

### 12.3 YmpBridge

YMM4連携用のYmpBridgeでは、Named PipeやYMM4側API/操作を使った直接反映が設計されている。

できること:

## 17. 提案: 1ライセンス共有 / マスターライセンス

ここからは現行機能ではなく、追加したいライセンス機能の提案。

ユーザーの要望:

- 買い切り購入者が、別PCでも使えるようにしたい。
- 購入者が、別の編集者にも使わせられるようにしたい。
- ただし、購入者のパスワードは教えたくない。
- 1ライセンスを同時に複数人が使えるわけではなく、誰かが使っている間は他の人は使えないようにしたい。
- つまり「同時1名まで使えるマスターライセンス」のような仕組みが欲しい。

### 17.1 機能名案

候補:

- マスターライセンス
- 編集者貸出ライセンス
- 共有ライセンス
- 1席同時利用ライセンス
- オーナー管理ライセンス
- チーム貸出ライセンス

販売上わかりやすい名前:

- 「マスターライセンス」
- 「編集者貸出機能」
- 「同時1名の共有利用」

おすすめの表現:

「購入者のパスワードを共有せず、登録した編集者や別PCにライセンスを貸し出せます。同時利用は1名までなので、編集者が利用中は購入者側ではPremium機能を使えません。」

### 17.2 目指すUX

購入者側:

- 買い切りライセンスを購入する
- 自分のアカウントにマスターライセンスが紐づく
- 別PCや編集者メールアドレスを招待できる
- 招待相手にパスワードは教えない
- 管理画面で現在の利用者を確認できる
- 必要ならライセンス貸出を取り消せる
- 必要なら自分の利用へ切り替えられる

編集者側:

- 自分のアカウントでログインする
- オーナーからの招待を受ける
- オーナーのパスワードを知らずにYMPを使える
- 利用できるのはオーナーが許可したライセンス枠
- オーナーまたは他編集者が使用中なら待機になる

競合時:

- 編集者が利用中のとき、オーナーが起動すると「現在、編集者Aがこのライセンスを利用中です」と表示する
- オーナーは待つ、利用リクエストを送る、強制切替する、のいずれかを選べる
- 強制切替した場合、編集者側には「オーナーがライセンスを回収しました」と表示する
- 編集中データの破損を避けるため、即時強制終了ではなく、Premium機能停止または保存猶予を設けるのが望ましい

### 17.3 基本ルール

ルール案:

- 1つの買い切りライセンスにつき、同時利用は1セッションまで
- オーナー本人のPCでも、編集者のPCでも利用できる
- ただし同時に使えるのは1人だけ
- パスワード共有は禁止
- 編集者は自分のアカウントでログインする
- オーナーは編集者を招待・削除できる
- オーナーは利用中セッションを取り消せる
- 編集者はライセンスの再販売、第三者共有、公開配布はできない
- 登録できる編集者数は上限を設ける
- 登録PC数にも上限を設けると不正利用を抑えやすい

おすすめ初期値:

- 同時利用: 1人
- 登録編集者: 2人から3人まで
- 登録端末: 3台から5台まで
- セッション維持: ハートビート方式
- オフライン猶予: 15分から30分程度

### 17.4 セッション貸出モデル

この仕組みは「ログイン情報の共有」ではなく「ライセンス利用権の一時貸出」として実装する。

概念:

- オーナーアカウント
- 購入済みマスターライセンス
- 招待された編集者アカウント
- 利用中セッション
- 期限付きリース
- ハートビート
- 失効/回収/強制切替

セッション取得の流れ:

1. ユーザーがログインする。
2. 自分のライセンスまたは招待された共有ライセンスを選ぶ。
3. サーバー側トランザクションで利用枠を取得する。
4. activeSession が空なら利用開始。
5. activeSession が存在し、期限切れなら上書きして利用開始。
6. activeSession が存在し、まだ有効なら利用不可。
7. 利用中は数分ごとにハートビートを送る。
8. アプリ終了・ログアウト時にreleaseする。
9. 通信断やクラッシュ時は leaseExpiresAt を過ぎたら自動解放する。

### 17.7 セキュリティ・プライバシー方針

必須:

- オーナーのパスワードを編集者に共有しない
- 編集者は自分のアカウントでログインする
- ライセンス貸出はサーバー側で管理する
- 生トークンを画面に表示しない
- deviceHashは復元不能なハッシュにする
- 表示名や端末名はマスクする

- 猶予後はPremium機能を停止または保存のみ許可
- オフライン中に別PCで同時利用できないようにする

理由:

- 買い切り共有ライセンスは便利だが、オフライン無制限にすると同時利用制限を回避されやすい。

- アップデート対応を継続する前提です。
- 個人制作者、まとめ動画運営者、外注編集者を使うチャンネル運営者に刺さる訴求にしてください。
- 高く売る場合、なぜその価格でも納得されるかを説明してください。
- 安く売る場合、サポートやアップデート範囲をどう制限すべきかも提案してください。

## 20. まとめ

YMP Ver.2 は、単なる台本生成ツールではなく、まとめ動画制作の前工程を広く支援する制作補助ソフト。

現行の強みは、対応サイト取得、AI台本作成、台本編集、サムネAI、YMM4テンプレ運用、外部AI/MCP連携、診断・サポート基盤までがつながっていること。

一方で、販売文面では「完全自動」ではなく「制作準備と反映を効率化」と言うべき。

追加予定のマスターライセンスは、買い切り版の価値を大きく上げられる。購入者がパスワードを共有せず、別PCや編集者にライセンスを貸し出せる。ただし同時利用は1名までにすることで、ユーザーの利便性と不正利用対策を両立できる。
